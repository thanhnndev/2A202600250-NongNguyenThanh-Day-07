### Cosine Similarity (Ex 1.1)

**High cosine similarity nghĩa là gì?**
> High cosine similarity (gần 1.0) nghĩa là hai vector embedding có hướng gần như song song nhau trong không gian đa chiều, tức là hai văn bản có ý nghĩa ngữ nghĩa rất gần nhau. Giá trị càng cao thì sự tương đồng về mặt ngữ nghĩa càng lớn.

**Ví dụ HIGH similarity:**
- Sentence A: "Laser eye surgery technology using femtosecond laser"
- Sentence B: "SmartSight laser system for refractive surgery"
- Tại sao tương đồng: Cả hai đều nói về công nghệ phẫu thuật laser mắt, sử dụng thuật ngữ chuyên ngành y tế liên quan đến điều trị tật khúc xạ.
- Actual Score: 0.7812 (rất cao)

**Ví dụ LOW similarity:**
- Sentence A: "Medical sterilization equipment for hospital use"
- Sentence B: "Phaco equipment for ophthalmic surgery procedures"
- Tại sao khác: Một câu nói về thiết bị tiệt trùng (sterilization), câu kia nói về thiết bị phẫu thuật nhãn khoa (phaco/ophthalmic). Hai lĩnh vực hoàn toàn khác nhau trong y tế.
- Actual Score: 0.3966 (thấp, dưới 0.5)

**Tại sao cosine similarity được ưu tiên hơn Euclidean distance cho text embeddings?**
> Cosine similarity được ưu tiên hơn vì nó đo lường góc giữa hai vector (hướng) thay vì khoảng cách tuyệt đối (magnitude). Văn bản có độ dài khác nhau có thể có ý nghĩa tương tự, cosine similarity sẽ phát hiện được sự tương đồng này trong khi Euclidean distance bị ảnh hưởng bởi độ dài văn bản. Ngoài ra, embeddings đã được normalized (chuẩn hóa) nên cosine tương đương với dot product, tính toán nhanh hơn.

### Chunking Math (Ex 1.2)

**Document 10,000 ký tự, chunk_size=500, overlap=50. Bao nhiêu chunks?**
> Phép tính:
> - Step size = chunk_size - overlap = 500 - 50 = 450
> - Chunk 1: 0-500 (500 chars)
> - Chunk 2: 450-950 (500 chars)
> - ...
> - Công thức: ceil((total_length - chunk_size) / step) + 1
> - Nếu overlap ở giữa các chunk: (10000 - 500) / 450 + 1 = 21.1 + 1 ≈ 22 chunks
> 
> Đáp án: **22 chunks**

**Nếu overlap tăng lên 100, chunk count thay đổi thế nào? Tại sao muốn overlap nhiều hơn?**
> - Step size mới = 500 - 100 = 400
> - Chunk count = (10000 - 500) / 400 + 1 = 23.75 + 1 ≈ 24 chunks
> - Chunk count tăng từ 22 lên 24 chunks
> 
> Tại sao muốn overlap nhiều hơn:
> 1. **Bảo toàn ngữ cảnh (context preservation)**: Đảm bảo câu/cụm từ bị cắt ở ranh giới chunk vẫn xuất hiện đầy đủ ở chunk tiếp theo
> 2. **Tránh mất thông tin quan trọng**: Thông tin nằm ở ranh giới giữa hai chunk không bị mất
> 3. **Tăng khả năng retrieval**: Query có thể match với nhiều chunks hơn nếu context được duplicate

### 4.2 Chunking Functions Implementation

#### `SentenceChunker.chunk` — approach:
> Tôi dùng regex `(?<=[.!?])\s+|\.\n` để detect sentence boundaries - pattern này match dấu câu (., !, ?) theo sau bởi khoảng trắng hoặc newline. Edge cases tôi xử lý:
> 1. **Empty sentences**: Filter `if s.strip()` để loại bỏ sentences chỉ có whitespace
> 2. **Whitespace stripping**: Dùng `s.strip()` cho mỗi sentence trước khi group
> 3. **Grouping**: Chia sentences thành groups theo `max_sentences_per_chunk`

```python
def chunk(self, text: str) -> list[str]:
    if not text:
        return []
    # Split on ". ", "! ", "? " or ".\n"
    sentences = re.split(r"(?<=[.!?])\s+|\.\n", text)
    # Filter out empty sentences and strip whitespace
    sentences = [s.strip() for s in sentences if s.strip()]

    chunks = []
    for i in range(0, len(sentences), self.max_sentences_per_chunk):
        chunk_sentences = sentences[i : i + self.max_sentences_per_chunk]
        chunk = " ".join(chunk_sentences)
        chunks.append(chunk.strip())

    return chunks
```

#### `RecursiveChunker.chunk` / `_split` — approach:
> Algorithm hoạt động đệ quy (recursive) với base case là `len(current_text) <= chunk_size`. Nếu text fit trong chunk_size, return ngay. Nếu không, thử tách bằng separator đầu tiên trong priority list. Với mỗi phần tách được, nếu phần đó vẫn quá lớn, gọi đệ quy `_split` với separators còn lại.
> 
> Priority separators: `["\n\n", "\n", ". ", " ", ""]` - từ coarse-grained đến fine-grained.

```python
def _split(self, current_text: str, remaining_separators: list[str]) -> list[str]:
    # Base case: if text fits in chunk_size, return it
    if len(current_text) <= self.chunk_size:
        return [current_text] if current_text else []

    # If no separators left, split by character
    if not remaining_separators:
        chunks = []
        for i in range(0, len(current_text), self.chunk_size):
            chunk = current_text[i : i + self.chunk_size]
            if chunk:
                chunks.append(chunk)
        return chunks

    separator = remaining_separators[0]
    next_separators = remaining_separators[1:]

    # Split by current separator
    parts = current_text.split(separator)
    chunks = []
    current_chunk = ""

    for part in parts:
        separator_prefix = separator if current_chunk else ""
        potential_chunk = current_chunk + separator_prefix + part

        if len(potential_chunk) <= self.chunk_size:
            current_chunk = potential_chunk
        else:
            if current_chunk:
                chunks.append(current_chunk)
            # If part itself is too big, recursively split with next separator
            if len(part) > self.chunk_size:
                sub_chunks = self._split(part, next_separators)
                chunks.extend(sub_chunks)
                current_chunk = ""
            else:
                current_chunk = part

    # Don't forget the last chunk
    if current_chunk:
        chunks.append(current_chunk)

    return chunks
```

### 4.3 EmbeddingStore

#### `add_documents` + `search` — approach:
> 
> **add_documents**: Mỗi Document được embed thành vector, sau đó lưu vào store dưới dạng record chứa id, content, embedding, và metadata. Hỗ trợ cả ChromaDB (nếu available) và in-memory fallback.
>
> **search**: Embed query, sau đó tính dot product (cosine similarity cho normalized vectors) với tất cả stored embeddings. Sort by score descending và return top_k.

```python
def _make_record(self, doc: Document) -> dict[str, Any]:
    record_id = f"{self._collection_name}_{self._next_index}"
    self._next_index += 1
    return {
        "id": record_id,
        "content": doc.content,
        "embedding": self._embedding_fn(doc.content),
        "metadata": {**doc.metadata, "doc_id": doc.id},
    }

def _search_records(
    self, query: str, records: list[dict[str, Any]], top_k: int
) -> list[dict[str, Any]]:
    query_embedding = self._embedding_fn(query)
    
    # Compute similarity for each record
    scored_records = []
    for record in records:
        score = _dot(query_embedding, record["embedding"])
        scored_records.append({**record, "score": score})
    
    # Sort by score descending
    scored_records.sort(key=lambda x: x["score"], reverse=True)
    return scored_records[:top_k]
```

#### `search_with_filter` + `delete_document` — approach:
> 
> **search_with_filter**: Filter trước khi search - lọc records theo metadata_filter (key-value matching), sau đó chỉ search trên subset đã lọc. Điều này hiệu quả hơn search rồi filter sau.
>
> **delete_document**: Remove tất cả chunks có `metadata.doc_id` matching với doc_id cần xóa. Return True nếu có xóa được ít nhất 1 record.

```python
def search_with_filter(
    self, query: str, top_k: int = 3, metadata_filter: dict = None
) -> list[dict]:
    if metadata_filter is None:
        return self.search(query, top_k)
    
    # Filter records by metadata first
    filtered_records = []
    for record in self._store:
        metadata = record.get("metadata", {})
        matches = all(metadata.get(k) == v for k, v in metadata_filter.items())
        if matches:
            filtered_records.append(record)
    
    return self._search_records(query, filtered_records, top_k)

def delete_document(self, doc_id: str) -> bool:
    initial_len = len(self._store)
    self._store = [
        record
        for record in self._store
        if record.get("metadata", {}).get("doc_id") != doc_id
    ]
    return len(self._store) < initial_len
```

### 4.4 KnowledgeBaseAgent

#### `answer` — approach:
> 
> Triển khai RAG pattern đơn giản:
> 1. **Retrieve**: Search top_k chunks từ EmbeddingStore
> 2. **Build context**: Join các chunks với `\n\n` separator
> 3. **Construct prompt**: Template có chỗ cho context và question
> 4. **Generate**: Gọi `llm_fn(prompt)` để sinh answer

```python
def answer(self, question: str, top_k: int = 3) -> str:
    # Retrieve top_k chunks from store
    results = self._store.search(question, top_k=top_k)
    
    # Build context from retrieved chunks
    context = "\n\n".join([r["content"] for r in results])
    
    # Build RAG prompt
    prompt = f"""Based on the following context, please answer the question.

Context:
{context}

Question: {question}

Answer:"""
    
    # Call the LLM function
    answer = self._llm_fn(prompt)
    
    return answer
```

## Phụ lục: Giải thích Retrieval Metrics

### A.1 Cơ bản Metrics (from EVALUATION.md)

**Retrieval Precision**:
- Top-k Relevance: Bao nhiêu kết quả trong top-k thực sự liên quan
- Benchmark Score: Mỗi query 2 điểm — top-3 relevant + answer chính xác = 2

**Chunk Coherence**: Chunk có giữ nguyên ý hoàn chỉnh hay bị cắt giữa câu/giữa ý?

**Metadata Utility**: Filter theo metadata có cải thiện kết quả không?

**Grounding Quality**: Agent answer có dựa trên context retrieved hay không?

### A.2 Advanced Metrics (BEIR Scifact)

**Recall@k** = |{relevant} ∩ {retrieved@k}| / |{relevant}|
- Ý nghĩa: Tìm được bao nhiêu % tài liệu relevant trong tổng số relevant
- Khi nào tốt: Cần đảm bảo không bỏ sót thông tin (comprehensive search)
- Ví dụ: Recall@5 = 0.80 nghĩa là tìm được 80% tài liệu relevant trong top-5

**Precision@k** = |{relevant} ∩ {retrieved@k}| / k
- Ý nghĩa: Trong kết quả trả về, bao nhiêu % là thực sự relevant
- Khi nào tốt: Cần kết quả chính xác, ít nhiễu (high-quality results)
- Ví dụ: Precision@5 = 0.32 nghĩa là 32% trong top-5 là relevant

**MRR@k** (Mean Reciprocal Rank) = 1 / (vị trí của relevant doc đầu tiên)
- Ý nghĩa: Đánh giá vị trí của kết quả relevant đầu tiên
- Khi nào tốt: Quan trọng vị trí đầu tiên relevant (user chỉ xem top-1 hoặc top-2)
- Ví dụ: MRR@5 = 0.7333 nghĩa là relevant doc đầu tiên ở vị trí ~1.36 (1/1.36 ≈ 0.7333)

**MAP@k** (Mean Average Precision) = Trung bình của precision@i tại các vị trí relevant
- Ý nghĩa: Tổng hợp ranking quality trên tất cả các relevant docs
- Khi nào tốt: Cần đánh giá tổng thể ranking quality, không chỉ top-1
- Công thức: MAP = (P@1 + P@2 + ... + P@k_relevant) / |{relevant}|
- Ví dụ: MAP@5 = 0.6500 cho thấy ranking chất lượng tốt

**NDCG@k** (Normalized Discounted Cumulative Gain)
- Ý nghĩa: Xếp hạng có trọng số, vị trí cao quan trọng hơn vị trí thấp
- Khi nào tốt: Thứ tự ranking quan trọng (relevant doc ở #1 tốt hơn ở #5)
- Công thức: 
  - DCG = Σ (rel_i / log2(i+2)) — gain giảm dần theo vị trí
  - NDCG = DCG / ideal DCG — normalize so sánh giữa các queries
- Ví dụ: NDCG@5 = 0.7619 cho thấy ranking gần ideal (1.0 = perfect ranking)

### A.3 So sánh Metrics

| Metric | Focus | Ưu điểm | Nhược điểm |
|--------|-------|---------|------------|
| Recall@k | Coverage | Không bỏ sót | Có thể nhiều nhiễu |
| Precision@k | Accuracy | Kết quả chất lượng | Có thể bỏ sót |
| MRR@k | First relevant | Đơn giản | Chỉ quan tâm top-1 |
| MAP@k | All relevant | Toàn diện | Phức tạp hơn |
| NDCG@k | Ranked relevance | Position-aware | Cần graded relevance |

**Khi nào dùng metric nào?**
- **Medical search** → High Recall (không bỏ sót treatment options)
- **Fact-checking** → High Precision (chỉ trả về facts chính xác)
- **Recommendation** → High MRR/NDCG (top results quan trọng nhất)
- **Research retrieval** → High MAP (tất cả relevant docs đều quan trọng)

### A.4 BEIR Benchmark Importance

**BEIR (Benchmarking IR)**:
- Tập hợp 18 diverse retrieval datasets
- Zero-shot evaluation: Train trên một dataset, test trên dataset khác
- Đánh giá khả năng generalize của retriever
- Scifact: Scientific fact verification (Y khoa/khoa học)

**Tại sao BEIR quan trọng cho evaluation?**
1. **Ground truth có sẵn**: Qrels (query relevance judgments) cho phép tính metrics khách quan
2. **Standardized**: Cộng đồng IR công nhận, so sánh được với state-of-the-art
3. **Diverse**: Nhiều domain khác nhau (science, news, FAQ, bio-medicine)
4. **Real-world**: Các queries và documents từ thực tế, không synthetic

**Key insight from BEIR evaluation**:
> Khi corpus size tăng lên (từ 5K đến 5M documents), sự khác biệt giữa các retrievers converge. Tuy nhiên, với medium-sized corpora (5K-50K docs), choice of chunking strategy và embedding model tạo ra sự khác biệt đáng kể (20-30% NDCG improvement).
